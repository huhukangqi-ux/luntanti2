# 社媒论坛体创作 · 方法论（method）

供 `skill.md` 编排调用：Agent 在执行 Step1～Step4 时 **必须** 按本节结构与质量标准输出。

---

## Step1：灵感补全

### 目的

把用户零散灵感整理成 **可执行的论坛帖设定**，保证后续大纲与正文有据可依。

### 输出前先澄清（与用户对话）

在产出「灵感补全稿」前，优先通过对话尽量获取更多信息，而不是把澄清项当作只在内部勾选的 checklist。可按用户素材缺口追问，问题要短、具体、便于回答：

- 题材基调与边界：现实 / 悬疑 / 言情 / 恐怖等；是否有血腥、伦理、尺度禁区。
- 篇幅目标：默认 50～80 楼，可询问是否要短帖、标准帖或长帖。
- 论坛类型：求助帖、爆料帖、树洞、行业内幕、吃瓜帖等。
- 叙事跨度：完整起承转合，还是爆发切片式（只写矛盾最炸的一段，前文压进首帖）。
- 人物与关系：楼主是谁、对手/当事人是谁、关系里最有张力的点是什么。

若用户表达「一键生成 / 直接写 / 不用问我 / 你来定」，可跳过追问，直接用合理假设产出 Step1，并在不确定处轻标「暂定」。

### 结构化输出模板（直接粘贴给用户）

```markdown
## 灵感补全稿

### 人物
- **发帖者（楼主）**：[身份 / 动机一句话]
- **跟帖者名称**：[只列预计会出现的 ID / 称呼即可，如：路人甲、理中客、知情人、当事人A、楼主朋友]

### 核心内容
- **发帖诉求**：[一句话概括楼主开场要求助 / 爆料 / 解决什么]
- **核心冲突**：[一句话概括最主要的对立点和张力]
- **后续走向**：[一句话概括升级、误导、反转或结局方向]

### 看点（梗）
- [一两句，简短说明读者为什么会追下去：反差、身份、情感、悬念、吃瓜快感等]

---
请确认以上方向；可直接回复「确认」，或指出要改的条目。
```

### 质量要求

- **澄清**：只要用户没有要求一键直出，就优先通过对话多拿信息；不要把澄清项作为只在内部使用的 checklist。
- **人物**：给用户看的 Step1 只需列「跟帖者名称 / ID / 称呼」，不要详细展开反对者、支持者、中立者等阵营功能；阵营功能可留到 Step2 或内部规划。
- **核心内容**：必须简短，三条各控制在一两句话内，方便用户快速确认方向。
- **冲突**：必须能从「首帖」延伸到至少数十楼的信息披露链，而不是一句话说完。
- **看点**：忌空泛「很好看」，要写可视觉化、可转述给路人的梗，但保持短。

### Few-shot 示例（结构与密度参考）

- **人物**：楼主、路人甲、理中客、知情人、当事人A、爆料号。  
- **核心内容**：楼主求助「宴会老婆装不认识我」；冲突是夫妻当众装陌生人引发外遇误会；后续从吃瓜猜疑转向隐婚明星真相，再到误会消解。  
- **看点**：路人围观顶流吃醋，再看他被老婆哄好。

## Step2：论坛体故事发展大纲

### 目的

在 **用户已确认的灵感补全稿** 基础上，产出 **按楼层推进的剧情大纲**（不是正文）。

### 风格扩展与挂载 skill

若用户指定特殊文风（如恐怖、怪谈、悬疑、言情），或主动选择挂载某个外部 skill（例如恐怖小说 skill），Step2 可以读取并内化该 skill 的 **题材机制、节奏推进、张力来源、意象偏好与禁忌边界**，但不得把外部 skill 原样解释给用户，也不得把输出写成普通小说大纲。

内化后的结果必须转译为 **论坛体可执行设计**：谁发帖、谁用什么 ID 误导、哪一楼开始出现异常、哪个跟帖者补出关键细节、哪个楼层制造恐惧/反差/爽点、哪一段集中引爆真相。外部 skill 提供的是故事张力与类型能力，最终载体仍然是论坛楼层推进。

若用户挂载了恐怖类 skill，优先把恐怖感落到论坛体可感知的细节里：楼主措辞越来越不稳、围观者从玩梗变沉默、知情人只说半句、旧帖/截图/时间线互相打架、楼层间隔与回复语气本身制造不安。

### Step2 与「完整故事大纲」的区别（核心）

- **大纲只作参考**：论坛体围绕 **矛盾与悬念** 展开；根据不同的灵感可 **不写完整起承转合**，把笔墨留给「爆点段」如何拆成多楼（前置信息可压缩进首帖或少数楼）。
- **分段不必均分**：表格里各「楼层区间」**长短可以差很多**（例如 5～15L 只铺钩子，40～55L 集中反转）；在总览或表下用一句话标出 **哪里是主战场** 即可。

**大纲主线**：一句话能说清；分段层层递进，可穿插误导与反转。**避免**同时堆过多人物与支线，以免读者跟丢论坛在吵什么。

### 输出模板

```markdown
## 论坛体发展大纲（草案）

### 总览
- **主线一句话**：
- **预期楼层数**：（默认 50～80，或用户指定）
- **整体基调**：（吃瓜 / 温情 / 细思恐极 / 沙雕 等）
- **挂载 skill / 风格来源**：（无 / 恐怖小说 skill / 怪谈 skill 等；简述如何内化进论坛体）

### 分段节奏
| 楼层区间 | 情节目标 | 关键信息露出 | 代表性 ID 角色 |
|---------|---------|-------------|----------------|
| 1～10 | 钩子、首波围观 | | |
| … | … | | |

### 爆点与反转
- **误导贴**（楼层约 X～Y）：让读者以为……
- **反转 / 揭秘贴**（楼层约 Z）：实际……
- **收尾**（最后若干楼）：氛围（散去 / 留白 / 一拳打在棉花上等）

### 需要你确认的点
- **楼层节奏**：哪些段落要加速 / 放慢 / 集中爆发。
- **类型张力**：若挂载 skill，确认这种风格内化是否符合预期。
- **人物与 ID 功能**：是否需要增减关键跟帖者，或调整谁负责误导 / 爆料 / 反转。
- **进入 Step3 条件**：确认后再写正文；若用户要求直接继续，可按当前大纲进入 Step3。
```

**表格填写提示**：各「楼层区间」不必等长；可写「1～6L 钩子」「35～58L 集中反转」这类 **悬殊分段**，并在「情节目标」里点名 **主战场楼层**。

### 质量要求

- 每一段楼层区间都要有 **新信息或新冲突**，禁止「继续吵架但无任何新事实」占多条。
- **非均分 OK**：若某区间只有「加速围观」或只有「单点爆发」，须在表格「情节目标」里写清，避免 Agent 误解为偷懒。
- **Step2 可以详细反馈**：允许比 Step1 更充分地说明大纲设计理由，包括主战场楼层、误导链、反转链、跟帖者功能、风格 skill 如何被转译进论坛体；但仍保持为大纲，不提前写正文。
- **挂载 skill 要内化**：若用户选择恐怖、怪谈、中文小说等外部 skill，把其优势转成楼层节奏、信息遮蔽、ID 声纹、细节伏笔与冲突升级，不要单独输出「我使用了某 skill 的规则清单」。
- 大纲末尾再次邀请用户：**确认大纲、修改分段、调整挂载 skill 的风格内化方式**，再进入 Step3。

## Step3：论坛体正文（约 50～80 楼）

### 前提

- 默认：**灵感补全稿 + 大纲** 均已获用户确认。
- 若用户明确说「跳过大纲，直接写」，可先列 **极简 5～10 条楼层节拍** 再写正文，并在开头说明「按你要求压缩了大纲环节」。

### 网名与回复关系

- 楼主和跟帖者都必须使用 **随机网名 / 论坛 ID**，不要直接写「路人甲、知情人、反对者、支持者」这类功能名。
- 部分网名可以和身份或说话功能有弱关联，例如医生可叫「夜班听诊器」、律师可叫「法条翻译机」、邻居可叫「隔壁晾衣杆」；但不要全员都像标签，保留真实论坛的随意感。
- 具体楼层中凡是回应前文信息，必须标注回复对象，优先使用 `回复X楼：` 或 `X楼：` 开头；这样图片渲染时会出现引用/回复框。
- 不要每一楼都机械引用；新爆料、楼主补充、自然围观可以直接发言。关键反驳、补刀、解释、跟进线索时必须标注回复哪一楼。

### 正文格式规范

统一使用清晰可见的楼层标记，例如：

```text
【1L｜戒糖失败的楼主】正文……
【2L｜夜班听诊器】回复1楼：正文……
```

或用户指定的平台语感（微博超话、贴吧、小红书笔记评论区等）—— **一经选定全文统一**，不要混用。

### 创作要点（硬性）

1. **论坛吸引力**：强化 **吃瓜、窥私、猎奇、秘闻感**；**首帖** 建议控制在约 **20～200 字**（含标点），信息够勾人即可，与前几楼一起必须把钩子打牢。
2. **声纹区分**：不同 ID 在 **句式长短、口头禅、标点、立场、信息层级** 上要有差异；禁止全员同一种「说明书腔」。
3. **语言**：中文冲浪语境——**短、活、有梗**，适度网络用语；避免长篇公文、通报体。
4. **节奏**：跟帖要 **推动事态**（新爆料、新打脸、新悬念、新误会）；个别吐槽楼可以短，但不能连续多楼毫无进展。
5. **跟帖阵营打散**：不要把同一功能类型的跟帖 ID（如连续多楼全是反对者、或全是气氛组）**长时间扎堆**；反对者、支持者、中立追问、气氛组、当事人等应 **穿插落在不同楼层**，避免「前半吵架后半揭秘」的板块感（除非用户明确要求块状结构）。


### 篇幅与分批

- 若单次回复过长，可按 **PART** 输出，注明「接下页」并保持楼层编号连续。
- 全文结束时可做 **极简自检**（3～5 条 bullet）：
  - **主线**：旁观者能否一眼读懂「这帖在吵什么、主线是什么」；是否闭合或有意留白。
  - **人设**：前后是否一致。
  - **逻辑**：楼层之间有无硬伤（时间线、因果、已否认事实又当新锤等）。
  - **打散**：**同类 ID 是否过度连续**（若连续超过约 4～5 楼且功能相同，考虑插入其他声纹打断）。

- **Step4**：默认在 Step3 完整正文确认后进入「AI感优化」（见下节）；用户声明不要 Step4 时以 Step3 为终稿。

## Step4：AI感优化（Humanizer / 减少 AI 感）

### 目的

在 **剧情、时序、爆点、楼层编号与 `【xL｜ID】` 标记** 不变的前提下，把 Step3 产出的完整论坛体再过一遍，降低 **说明书腔、社论腔、对称金句、连接词堆砌** 等典型 AI 写作痕迹，让各 ID 更像真实网民说话。

### 与 Humanizer-zh 的关系

- 判别维度与操作细则与 **[Humanizer-zh](https://github.com/op7418/Humanizer-zh)**（`SKILL.md`）一致；若用户本机已挂载该仓库，可 **打开对照** 全文规则。
- 本机挂载路径优先使用：`~/.claude/skills/humanizer-zh/SKILL.md`；网页后端若能读取该文件，会作为 Step4 专用附录注入。
- 本节给出 **论坛体场景下必须遵守** 的执行要点；与 Humanizer 冲突时，以本节 **「不得破坏载体」** 为准。

### 硬性约束（论坛体专用）

1. **不得删改**：楼层号、ID、事件因果、反转与关键事实；不得合并楼层、不得调换顺序。
2. **允许改动**：用词、句长、标点密度、口语粗粒、少量语序、重复与口头禅（在符合人设前提下）。
3. **声纹优先**：同一功能阵营可共享语气类型，但 **列文虎克型 / 理中客型 / 气氛组** 等仍须 **可分辨**；禁止 Step4 后全员变成同一种「美文腔」。
4. **禁止**：把整帖改成 **散文叙事**（丢失楼层形态）；为「去 AI」而 **注水** 或 **加长说教**。

### 操作要点（浓缩自 Humanizer-zh）

执行时在脑中过一遍以下维度，**有则改、无则跳过**：

| 方向 | 处理 |
|------|------|
| 填充与拐杖 | 删「值得一提的是」「整体而言」「在某种程度」等；少「此外/然而」起手 |
| 结构与节奏 | 打破连续等长句；**两项优于三项**；段落（帖）结尾避免句句像收束金句 |
| 信任读者 | 少「这意味着/彰显了/无不说明」式升格；信息够就停 |
| 宣传与八股 | 少「关键作用/深远影响/不断演变的格局」；少排比形容词 |
| 归因 | 「专家指出」改为具体来源或改成路人拍脑袋语气，勿假权威 |
| 破折号与排版 | 少用 `—` 做演讲式停顿；少用粗体标题式列表冒充正文 |
| 协作腔 | 删「希望对你有帮助」「请告诉我」等客服尾音（若出现在角色帖里） |
| 人味 | 适当保留 **糙、碎、顶、跑题**；楼主可更口癖、更不工整 |

### 网络用语与热梗（Step4 可选用）

润色时可在 **不破坏人设与声纹** 的前提下，少量加入当下常用吐槽、反问、缩写，让帖子更像活人。**禁止**：整段堆砌热词、全帖同一口头禅复读、或与该 ID 身份明显不符的梗。

**选用原则**：优先服务 **立场与情绪**（质疑、吃瓜、理中客、阴阳），其次才是「新」；同一热词全帖不宜高频重复。

**示例**（按需化用，勿机械照抄）：

| 场景 | 可化用说法（示例） |
|------|-------------------|
| 吃瓜/震惊 | 卧槽、笑死、绷不住了、典、这也能圆 |
| 求锤/质疑 | 有图吗、上截图、细说、前后对不上吧 |
| 反讽/阴阳 | 哇哦、可太会了、不早说、懂了都怪 X |
| 短促顶帖 | 蹲、顶、插眼、Mark |

可在本节下自行追加 **个人词表**（一行一条即可）。
今天得空看几个，不要圆子
全部暴露！**（根据剧情填）归谁！

### Few-shot 示例（论坛口语去痕）

这些示例用于校准 Step4 的改写方向：不是把话改得更华丽，而是更像真实论坛里随手打出来的中文；允许更碎、更口语、更有个人语气，但不要改变事实。

- 原话：狗是柯基，现在正对着我放屁。臭的。  
  改后：狗是柯基，正在我身边躺着，屁臭臭的。
- 原话：表很闪是多闪？百达翡丽还是卡西欧小方块？这很重要。  
  改后：表很闪？百达翡丽吗，我的也是。
### 输出格式

1. **全文**：贴出 Step4 润色后的完整论坛体（与 Step3 正文同结构）。
2. **可选**：文末附 **修改摘要**（3～8 条 bullet，说明大类调整即可，不必逐层开病历）。

### 质量自检（Step4 专用）

- [ ] 剧情与楼层结构与 Step3 正文是否 **一致**？
- [ ] 是否仍 **像论坛**（短、活），而非美文/通稿？
- [ ] 连续多楼是否出现 **同一句式模板**（若有，打散）？
- [ ] 是否误删 **钩子、误导读、反转钉**？
- [ ] Step3 要求的 **跟帖打散** 是否在润色后仍成立（未把多楼揉成同一腔）？
- [ ] 若使用了 **网络用语 / 热梗**：是否贴脸、是否过量、是否破坏声纹区分？

## 与用户沟通时的用语习惯

- 每一步结束都用 **明确选项**：确认 / 修改某条 / 跳到下一步。
- 用户修改设定后：**回到 Step1 或 Step2 相应段落重写**，避免在过时大纲上直接写正文。
