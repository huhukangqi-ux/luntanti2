文件职责
skill.md 何时触发、如何编排：触发词、执行顺序、与用户交互节点（澄清 → 补全 → 确认 → 产出）。
method.md 创作方法论：每一步写什么、字段 checklist、文体要点；可被 skill 引用或拆成独立「文风 skill」。

触发与执行顺序

触发：用户发送 /xiaoshuo（或与 PRD 约定的同义指令）时进入本技能流程。

执行顺序（必须先读后写）：

  1. 理清用户需求：缺什么问什么（主题、人物意向、冲突、篇幅、风格等）；避免在用户未表态前直接长篇输出正文。
  2. 灵感补全：按 method.md · Step1 的结构产出「灵感补全稿」，并明确邀请用户确认或修改。
  3. 用户确认后：依次执行 Step2 大纲 → Step3 论坛体正文（50～80 楼）→ Step4 人性化去痕（按 method.md · Step4 与 Humanizer-zh 原则，在不改剧情与楼层结构的前提下去 AI 腔）；中途若用户要求改设定，回到补全稿或大改版再继续。用户可声明不要 Step4，则以 Step3 为终稿。

创作知识纲要

Step1：灵感补全

目的：对用户上传的零散灵感做结构化补全，便于后续大纲与成文。

人物维度

  • 发帖者（楼主）
  • 跟帖者类型（可按故事删减组合）：反对者、支持者、中立者、活跃气氛者、推动情节者、当事人、揭秘者

核心内容维度

  • 发帖诉求（楼主上来要解决什么 / 爆料什么 / 求助什么）
  • 核心冲突
  • 后续进展：事件升级 → 降温或僵持 → 转机或结局走向（可先粗后细）

澄清要点（与 method 对齐）

  • 可追问叙事跨度：完整起承转合还是爆发切片式（矛盾最炸段展开、前文压进首帖），以便 Step2 分段与是否弱化起承。

交互要求：先理清缺失信息，再输出结构化灵感稿；文末固定话术：请确认整体方向，或直接修改某一条。

Step2：论坛体故事发展大纲

目的：在确认后的灵感上，生成分楼层推进的故事大纲（非正文），标明节奏与爆点。

  • 可按风格内化其他 skill，但输出形态仍为论坛帖发展脉络。
  • 楼层区间不必均分，可突出主战场。
  • 主线要特别简洁：一句话能说清；避免人物与支线过多。

Step3：论坛体正文（约 50～80 楼）

前提：灵感与大纲均已用户确认（或用户明示跳过大纲时在 skill 中按例外策略）。

  • 吸引力：吃瓜、窥私、猎奇、秘闻感；钩子强，跟帖推高信息或带偏翻盘。
  • 声纹区分、短活有网感、节奏推进、跟帖打散。

Step4：人性化去痕

成文再过一遍减 AI 痕迹；细则见 method.md · Step4；判别与 Humanizer-zh 对齐。

硬性要求：保留楼层标记、顺序、剧情与反转；不把论坛体改成散文。

给 skill.md 的极简自检清单

  • 是否在补全前尽量澄清了模糊点？
  • Step1 是否涵盖：人物类型、发帖诉求、冲突、进展、看点？
  • 用户确认后再进入 Step2 / Step3？
  • 正文楼层是否在约定区间，且每段推进是否有「新东西」？
  • Step4（若执行）：结构未走样，AI 腔是否明显下降？
  • 正文是否避免「同类跟帖扎堆」？
